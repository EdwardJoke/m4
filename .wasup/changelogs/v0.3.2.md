## [v0.3.2] - 2026-06-25

### Added

- **`-D` flag for QBE optimization levels** — CLI now accepts `-Dfast` (fast compilation, larger binary) and `-Dsmall` (longer compilation, smaller binary) to control QBE's optimization pass. Passed through to `qbe_compile_ssa` in the native build pipeline
- **Unboxed integer representation in QBE backend** — Integer literals now emit `%r =l copy <value>` instead of heap-allocating via `$m4_new_int`. Binary arithmetic (add/sub/mul/div/mod) and comparisons emit native QBE instructions directly, eliminating allocation and runtime dispatch overhead for ints
- **Boxed/unboxed type tracking** — The QBE emitter tracks per-temp boxed (M4Value*) vs unboxed (raw `l`) status. Boxing calls (`$m4_box_int`) are automatically inserted at type boundaries: function calls, vec/struct stores, index/field access, and `m4_is_truthy` calls
- **Unboxed runtime entry points** — New `m4_add_u`, `m4_sub_u`, `m4_mul_u`, `m4_div_u`, `m4_mod_u`, `m4_neg_u`, `m4_not_u` for raw int arithmetic, alongside the existing boxed M4Value* paths
- **Unboxed if-condition optimization** — When the condition is an unboxed value, `jnz` is emitted directly instead of routing through `$m4_is_truthy`

### Changed

- **`m4_new_int` is now identity** — Returns the raw `l` value directly instead of heap-allocating an `M4Value*`. Internal runtime callers updated to use `m4_box_int` when boxed ints are needed (e.g., storing into vecs/structs)
- **`m4_new_bool`/`m4_new_char` → native `copy`** — Bool and char literals now emit `%r =l copy <value>` with no heap allocation
- **QBE wrapping API** — Updated `qbe_wrap.c`/`qbe_wrap.h` to pass optimization level to QBE compiler
- **Updated CLI help text** — Documents `-Dfast`/`-Dsmall` with descriptions
